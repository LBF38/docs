⚔️ 保护你的 API

除了提供一个流畅的用户登录系统，Logto 默认还集成了 API 资源 的授权服务，从而更好的保护你的 API 资源免受匿名身份的访问和侵害。让我们通过以下步骤，使用 Logto 来实现一个你自己的服务端资源权限保护系统。

在 Logto 管控制台内注册 API 资源​

Logto 服务可以从每一个授权请求中识别出所申请的目标 API 资源，并相应的颁发其专属的 access_token 令牌。

访问 Logto 管理控制台的 API 资源 页面。在列表中你会看到一个拥有 https://api.logto.io. 作为 API identifier 的内置 API 资源。该资源代表了所有注册在此路由地址下，Logto 服务所提供的管理 API。它将保证我们所有的 API 都设置了权限保护，仅具有管理员角色的 Logto 授权用户才可以访问。

接下来，点按 创建 API 资源 按钮，并输入一下信息，来注册一个你自己的 API 资源：

  • 一个可读性高的 API 名称，以更好的帮助你在未来能够快速识别此 API 资源
  • 一个全局唯一的， URI 格式的，API Identifier (又可称之为 Resource Indicator)。用来识别受保护的 API 资源。


警告

API Identifier 将被 Logto 用作全局唯一的 API 资源标识符。 一经创建不可编辑。请小心填写。

创建后，新的 API 资源将显示在列表中。你可以过点按该 API 资源， 在 API 详情页面内对其进行管理或删除操作。

有关详细的 API 设置定义，请参阅 API 资源数据格式。

信息

所有在 Logto 管理控制台内注册的 API 资源信息可以被所有的应用所共享。

将资源授权流程集成到你的客户端应用程序中​

一旦在 Logto 服务器上成功地注册了你的 API 资源，我们就可以回到客户端应用程序中，来看看 Logto SDK 是如何工作的。

备注

如果你尚未将 Logto SDK 集成到你的客户端应用程序中，请阅读我们 SDK 的 集成指南。 Logto 提供了多种平台 SDK 来满足不同的客户端需求。专注于你的产品，让我们来处理繁琐的流程。

使用 Logto SDK，你只需在启动时将这些资源的 API Identifier 作为配置传给 SDK。

解析并验证 API 请求的权限令牌​

最关键的一步便是通过对一个可信赖的令牌进行验证来保护您的 API 资源。如上所述，所有经过授权的请求都应在其 Authorization 请求头中携带由 Logto 颁发的 JWT 格式的 assess_token 令牌。

获取 JWK 公钥​

Logto 颁发的所有令牌都使用了 JWK 签名。 （有关详细信息，请参阅 JWS）

在继续之前，我们需要获取一个公钥用来验证 Bearer Token (access_token) 的签名。

密钥集可以通过 Logto 的 jwks_uri 查询接口：https://<your-logto-domain>/oidc/jwks 来获取。

备注

所有最新的 Logto 授权服务器的配置都可以通过 https://<your-logto-domain>/oidc/.well-known/openid-configuration 找到，包括 jwks_uri 以及其他相关配置。

在返回数据中，你将获得一个 JWKS（JSON Web 密钥集）。 我们需要将其 保存 在你的服务端。它将被用来验证所有请求的令牌签名。

识别并从 Headers 中提取 Bearer Token​

一个经过授权的请求应包含有 Authorization 请求头，其格式为 Bearer <access_token>。 我们需要从请求头中提取出令牌：

令牌解析和验证​

一个经过编码的 JWS 令牌由三部分构成：

  • JOSE Header：声明令牌类型和编码算法
  • JWS Payload：包含所有令牌的内部参数
  • JWS Signature：使用 JWK 加密的签名

Logto JWS Payload 所包含的参数定义：

名称描述
jtiJWT 唯一 ID
sub对象, 一般为请求用户的 ID
iat代表令牌颁时间的时间戳
exp代表令牌过期时间的时间戳
client_id客户端应用 ID
iss令牌颁发方身份标识
aud令牌的受众资源标识

（根据不同的 OIDC 配置，JWS Payload 中还可以提供更多的扩展信息）

备注

在开发环境下，想要直观地检查 JWT 内容，可以通过访问 jwt.io 来解码并检查你所接收到的令牌。请小心使用任何生产环境中的令牌，因为此服务由第三方提供，你的令牌可能会被暴露。

一个合法且有效的令牌应满足一下条件：

  • 验证令牌的 JWS 格式有效性（有关详细信息，请参阅 JWT）
  • 令牌的颁发者是 https://<your-logto-domain>/oidc（由 Logto 服务器颁发）
  • 令牌的受众是当前接收令牌接口的 API Indicator（API 专属令牌）
  • 令牌在有效期内

不同平台中都提供了各种 JWT 相关的开源库和中间件。它们中的大多都提供了一个简单的公共方法用来帮助我们验证和解析 JWT 令牌。

  • NodeJs
  • Java

使用 jose 来验证令牌的签名，有效性，以及其所携带的授权信息。

import { jwtVerify } from 'jose';

const { payload } = await jwtVerify(
  token, // 从请求头中提取的 Bearer Token
  publicKey, // 从 Logto 服务器获取的存在本地的 jwks 公钥集
  {
    // 令牌的颁发者是 `https://<your-logto-domain>/oidc`
    issuer: 'https://<your-logto-domain>/oidc',
    // expected audience token, should be the resource indicator of the current API
    audience: '<your request listener>',
  }
);

JwtTokenUtil:

import java.util.Date;

import io.jsonwebtoken.Claims;
import io.jsonwebtoken.Jwts;

public class JwtTokenUtil {

    private final String secret;

    public JwtTokenUtil(String secret) {
        this.secret = secret;
    }

    public Boolean validateToken(String token, String listener, String logtoPath) {
        final Claims claims = getAllClaimsFromToken(token);
        final String audience = claims.getAudience();
        final String issuer = claims.getIssuer();

        return (audience.equals(listener) && issuer.equals(logtoPath) && !isTokenExpired(claims));
    }

    public Claims getAllClaimsFromToken(String token) {
        return Jwts.parser().setSigningKey(secret).parseClaimsJws(token).getBody();
    }

    public Date getExpirationDateFromToken(String token) {
        return getAllClaimsFromToken(token).getExpiration();
    }

    public String getUserIdFromToken(String token) {
        return getAllClaimsFromToken(token).getSubject();
    }

    public String getAudienceFromToken(String token) {
        return getAllClaimsFromToken(token).getAudience();
    }

    public String getIssuerFromToken(String token) {
        return getAllClaimsFromToken(token).getIssuer();
    }

    private static Boolean isTokenExpired(Claims claims) {
        return claims.getExpiration().before(new Date());
    }
}
